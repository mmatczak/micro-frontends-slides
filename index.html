How to implement this architecture?

Manfred Steyer

Hyperlinks

  • Extremely simple integration
  • Microservice UIs have to be bootstrapped each time when switching between them
  • Microservice UIs' state is lost, unless the UIs are able to store and and restore their state (e.g. Redux)

"Majestic Monolith"

  • Microservices UI's are npm packages and as such can be versioned and distributed; both mono-repo and multi-repo patterns are possible
  • All modules are integrated into one (Single Page) application at build time (no deployment at runtime possible)

Web Components...

...are a set of web platform APIs that allow you to create new custom, reusable, encapsulated HTML tags to use in web pages and web apps

webcomponents.org

What if...

...we exported the whole Microservice UI as a Web Component?

We could then easily integrate the Microservice UIs within the shell :)

@vue/vue-web-component-wrapper and Angular Elements allow to export Angular and Vue.js components as Web Components

<iframe>...

...represents a nested browsing context (having its own session history and document), embedding another HTML page into the current one

...can communicate with the parent browsing context (window) and other <iframe>s using Window.postMessage()

...has limited styling features, e.g. scrolling

<iframe>'s own browsing context...

...requires increased memory and other computing resources; BTW: how many browser tabs do you typically have open? :)

...requires a meta router to keep its session history in sync with the parent's one (to see URL changes or to add bookmarks)

How to put all these pieces together to work well for enterprises?

"Working well" means...

  • Microservice UIs are technologically independent
  • Microservice UIs use modern JavaScript frameworks making development efficient
  • Any shared components are implemented as Web Components to achieve interoperability

Keep in mind:

There is no one "right" solution.
